# Requirements Document

## Introduction

This feature adds email/password authentication to the EdgeMarket app (Phase 8). Every user must create an account with a verified email address before any screen is accessible. Email auth becomes the primary identity layer; the existing wallet-signature flow becomes a post-login "link wallet" action.

The feature covers:
- New PostgreSQL tables for users and email verification codes
- Five new Spring Boot endpoints for signup, email verification, code resend, login, and session restore
- React Native screens for signup, login, and email verification
- An `AuthNavigator` guard that blocks unverified users from the main app
- A `useAuth` hook that manages JWT lifecycle via AsyncStorage
- Migration of `/api/follows/*` and `/api/push-tokens/*` to derive identity from the JWT `sub` claim instead of a client-supplied `userAddress`

The existing wallet-signature-auth spec (Phase 7) remains intact. The two auth systems share the same `JWT_SECRET` and HS256 signing; they are distinguished only by the `sub` claim content (UUID for email auth, wallet address for wallet-signature auth).

---

## Glossary

- **User**: A registered account identified by a UUID, email address, and bcrypt password hash.
- **Email_Auth_Service**: The Spring Boot service responsible for user registration, verification code management, password checking, and JWT issuance for email-based accounts.
- **Verification_Code**: A 6-digit numeric string generated by `SecureRandom`, stored in `email_verification_codes`, valid for 10 minutes, single-use.
- **JWT**: An HS256-signed JSON Web Token with `sub = user UUID`, `iat`, and `exp = iat + 86400`. Uses the same `JWT_SECRET` and signing key as the wallet-signature-auth feature.
- **Auth_Guard**: The React Native navigator component that shows `AuthNavigator` (signup/login/verify screens) to unauthenticated or unverified users, and `RootNavigator` (main app) to authenticated users.
- **useAuth**: A new React Native hook that owns JWT storage in AsyncStorage (`@edgemarket/email-jwt`), exposes `signup`, `login`, `verifyEmail`, `resendCode`, `logout`, and `currentUser`, and provides the JWT to all API calls.
- **SendGrid**: The email delivery provider; called asynchronously via `SENDGRID_API_KEY`. Failures are logged but do not block the HTTP response.
- **Rate_Limit**: Maximum 3 resend-code requests per email per hour, enforced server-side.

## Requirements

### Requirement 1: User Registration (Signup)

**User Story:** As a new user, I want to create an account with my email and a password so that I can access the EdgeMarket app.

#### Acceptance Criteria

1. WHEN a `POST /api/auth/signup` request is received with a valid JSON body containing `email` and `password`, THE Email_Auth_Service SHALL create a new user row with `email_verified = false`, a bcrypt password hash (cost factor ≥ 10), a randomly generated UUID, and `created_at = NOW()`.
2. WHEN the user is created, THE Email_Auth_Service SHALL generate a 6-digit Verification_Code, persist it to `email_verification_codes` with `expires_at = NOW() + 10 minutes` and `consumed_at = NULL`, and send it to the user's email address via SendGrid asynchronously.
3. WHEN the signup succeeds, THE Email_Auth_Service SHALL return HTTP 201 with body `{ "message": "Verification code sent to your email" }`.
4. IF the `email` field is absent, empty, or does not match a basic RFC-5322-style pattern, THEN THE Email_Auth_Service SHALL return HTTP 400 with body `{ "error": "Valid email is required" }`.
5. IF the `password` field is absent or fewer than 8 characters, THEN THE Email_Auth_Service SHALL return HTTP 400 with body `{ "error": "Password must be at least 8 characters" }`.
6. IF an account with the same email address already exists (regardless of verification status), THEN THE Email_Auth_Service SHALL return HTTP 409 with body `{ "error": "Email already registered" }`.
7. THE Email_Auth_Service SHALL NEVER log the plaintext password at any log level.
8. THE Email_Auth_Service SHALL store only the bcrypt hash; the plaintext password SHALL NOT be persisted anywhere.

---

### Requirement 2: Email Verification

**User Story:** As a registered user, I want to verify my email address by entering the 6-digit code I received so that I can activate my account and receive a JWT.

#### Acceptance Criteria

1. WHEN a `POST /api/auth/verify-email` request is received with a valid JSON body containing `email` and `code`, THE Email_Auth_Service SHALL look up the most recent unconsumed, unexpired Verification_Code for the given email.
2. WHEN the code matches and is valid, THE Email_Auth_Service SHALL set `email_verified = true` on the user row, set `consumed_at = NOW()` on the code row (in the same database transaction), issue a JWT with `sub = user_id` (UUID), and return HTTP 200 with body `{ "token": "<jwt>" }`.
3. IF the code does not match, is already consumed, or has expired, THEN THE Email_Auth_Service SHALL return HTTP 400 with body `{ "error": "Invalid or expired code" }`.
4. IF no user with the given email exists, THEN THE Email_Auth_Service SHALL return HTTP 400 with body `{ "error": "Invalid or expired code" }` (same response as a wrong code, to avoid email enumeration).
5. IF the `email` or `code` fields are absent or empty, THEN THE Email_Auth_Service SHALL return HTTP 400 with body `{ "error": "email and code are required" }`.

---

### Requirement 3: Resend Verification Code

**User Story:** As a registered but unverified user, I want to request a new verification code if mine expired or I didn't receive it.

#### Acceptance Criteria

1. WHEN a `POST /api/auth/resend-code` request is received with a valid JSON body containing `email`, THE Email_Auth_Service SHALL invalidate all existing unconsumed codes for that email by setting their `consumed_at = NOW()`, generate a new 6-digit Verification_Code, persist it, and send it via SendGrid asynchronously.
2. WHEN the resend succeeds, THE Email_Auth_Service SHALL return HTTP 200 with body `{ "message": "New code sent" }`.
3. IF the email has had 3 or more resend requests within the last 60 minutes, THE Email_Auth_Service SHALL return HTTP 429 with body `{ "error": "Too many resend requests. Try again later." }` and SHALL NOT send a new code.
4. IF no user with the given email exists, THE Email_Auth_Service SHALL return HTTP 200 with body `{ "message": "New code sent" }` (same response to avoid email enumeration).
5. IF the `email` field is absent or empty, THE Email_Auth_Service SHALL return HTTP 400 with body `{ "error": "email is required" }`.
6. IF the user's email is already verified, THE Email_Auth_Service SHALL return HTTP 400 with body `{ "error": "Email already verified" }`.

---

### Requirement 4: Login

**User Story:** As a verified user, I want to log in with my email and password so that I receive a JWT to access the app.

#### Acceptance Criteria

1. WHEN a `POST /api/auth/login` request is received with a valid JSON body containing `email` and `password`, THE Email_Auth_Service SHALL look up the user by email and verify the password against the stored bcrypt hash.
2. WHEN the credentials are correct and `email_verified = true`, THE Email_Auth_Service SHALL issue a JWT with `sub = user_id` (UUID) and return HTTP 200 with body `{ "token": "<jwt>" }`.
3. WHEN the credentials are correct but `email_verified = false`, THE Email_Auth_Service SHALL return HTTP 403 with body `{ "error": "Email not verified. Check your inbox for the verification code." }`.
4. IF the email does not exist OR the password is wrong, THE Email_Auth_Service SHALL return HTTP 401 with body `{ "error": "Invalid credentials" }` in both cases (no email enumeration).
5. IF the `email` or `password` fields are absent or empty, THE Email_Auth_Service SHALL return HTTP 400 with body `{ "error": "email and password are required" }`.
6. THE Email_Auth_Service SHALL NEVER log the plaintext password submitted in a login request.

---

### Requirement 5: Session Restore

**User Story:** As a logged-in user, I want the app to restore my session on re-open so I do not need to log in every time.

#### Acceptance Criteria

1. WHEN a `GET /api/auth/me` request is received with a valid Bearer JWT, THE Email_Auth_Service SHALL return HTTP 200 with body `{ "id": "<uuid>", "email": "<email>", "emailVerified": true, "walletAddress": "<address_or_null>" }`.
2. WHEN the JWT is absent, expired, or invalid, THE Email_Auth_Service SHALL return HTTP 401 with body `{ "error": "Unauthorized" }`.
3. WHEN the app initialises, THE useAuth hook SHALL attempt to rehydrate the JWT from AsyncStorage key `@edgemarket/email-jwt`, validate its `exp` claim client-side, and if valid, call `GET /api/auth/me` to confirm the session before setting auth state to `authenticated`.
4. IF `GET /api/auth/me` returns 401 during rehydration, THE useAuth hook SHALL clear the stored JWT and set auth state to `unauthenticated`.
5. THE `GET /api/auth/me` endpoint SHALL be protected by the existing `AuthFilter`; the `sub` claim is the user UUID and `AuthFilter` SHALL set it as the `authenticatedUserId` request attribute.

---

### Requirement 6: Logout

**User Story:** As a logged-in user, I want to log out so that my session is cleared from the device.

#### Acceptance Criteria

1. WHEN `logout()` is called on the useAuth hook, THE hook SHALL remove the JWT from AsyncStorage (`@edgemarket/email-jwt`) and set auth state to `unauthenticated`.
2. WHEN `logout()` is called, THE hook SHALL clear `currentUser` from memory.
3. After logout, THE Auth_Guard SHALL redirect the user to the login screen.

---

### Requirement 7: Auth Guard and Navigation

**User Story:** As an unauthenticated or unverified user, I want to be blocked from the main app screens so that only verified users can use EdgeMarket.

#### Acceptance Criteria

1. WHEN the app renders, THE Auth_Guard SHALL check auth state and render `AuthNavigator` if the user is unauthenticated or `RootNavigator` if the user is authenticated with a verified email.
2. `AuthNavigator` SHALL contain three screens: `LoginScreen`, `SignupScreen`, and `VerifyEmailScreen`.
3. WHEN the user successfully verifies their email on `VerifyEmailScreen`, THE Auth_Guard SHALL automatically transition to `RootNavigator` without any manual navigation call.
4. WHEN the user logs out, THE Auth_Guard SHALL immediately render `AuthNavigator` and show `LoginScreen`.
5. THE Auth_Guard SHALL show a neutral loading indicator while the useAuth hook is rehydrating (before auth state is known), to prevent a flash of the login screen for users with a valid session.

---

### Requirement 8: SignupScreen

**User Story:** As a new user, I want a signup screen to create my account.

#### Acceptance Criteria

1. `SignupScreen` SHALL contain email and password text inputs and a "Create Account" button.
2. WHEN the user taps "Create Account", THE screen SHALL call `signup(email, password)` from useAuth, show a loading state during the request, and navigate to `VerifyEmailScreen` on HTTP 201.
3. WHEN the server returns HTTP 409, THE screen SHALL display "An account with this email already exists."
4. WHEN the server returns HTTP 400 for invalid email or password, THE screen SHALL display the `error` field from the response body.
5. THE password input SHALL use `secureTextEntry={true}`.
6. `SignupScreen` SHALL include a link or button to navigate to `LoginScreen`.

---

### Requirement 9: LoginScreen

**User Story:** As a returning user, I want a login screen to sign in.

#### Acceptance Criteria

1. `LoginScreen` SHALL contain email and password text inputs and a "Sign In" button.
2. WHEN the user taps "Sign In", THE screen SHALL call `login(email, password)` from useAuth, show a loading state, and on success store the JWT and transition to the main app via the Auth_Guard.
3. WHEN the server returns HTTP 401, THE screen SHALL display "Invalid email or password."
4. WHEN the server returns HTTP 403 (email not verified), THE screen SHALL display the server's error message and offer a "Resend Code" button that navigates to `VerifyEmailScreen` with the email pre-filled.
5. THE password input SHALL use `secureTextEntry={true}`.
6. `LoginScreen` SHALL include a link or button to navigate to `SignupScreen`.

---

### Requirement 10: VerifyEmailScreen

**User Story:** As a registered but unverified user, I want to enter my verification code to activate my account.

#### Acceptance Criteria

1. `VerifyEmailScreen` SHALL display the user's email address (passed as navigation param or read from useAuth state) and a 6-digit code input.
2. WHEN the user submits the code, THE screen SHALL call `verifyEmail(email, code)` from useAuth, show a loading state, and on success the Auth_Guard will automatically transition to the main app.
3. WHEN the server returns HTTP 400, THE screen SHALL display "Invalid or expired code. Please try again."
4. `VerifyEmailScreen` SHALL include a "Resend Code" button that calls `resendCode(email)` from useAuth.
5. WHEN `resendCode` returns HTTP 429, THE screen SHALL display "Too many resend requests. Try again later."
6. WHEN `resendCode` returns HTTP 200, THE screen SHALL display "A new code has been sent to your email."

---

### Requirement 11: useAuth Hook

**User Story:** As a developer, I want a single hook that encapsulates all email auth state so that screens don't manage JWTs directly.

#### Acceptance Criteria

1. THE useAuth hook SHALL expose: `signup(email, password)`, `login(email, password)`, `verifyEmail(email, code)`, `resendCode(email)`, `logout()`, `currentUser` (nullable object with `id`, `email`, `emailVerified`, `walletAddress`), `authState` (`'loading' | 'unauthenticated' | 'authenticated'`), and `error` (nullable string).
2. THE JWT SHALL be persisted to and loaded from AsyncStorage under key `@edgemarket/email-jwt`.
3. On mount, THE hook SHALL rehydrate: load the JWT, validate its `exp`, and if valid call `GET /api/auth/me`. It SHALL set `authState = 'loading'` during rehydration and resolve to `'authenticated'` or `'unauthenticated'` before any screen renders.
4. THE hook SHALL expose a `getJwt()` function (returns `string | null`) for use with `apiRequest()` in other hooks.
5. WHEN `authState` transitions to `'authenticated'`, THE hook SHALL populate `currentUser` from the `/api/auth/me` response.
6. THE hook SHALL be provided via a React context (`AuthContext`) so that `useAuth()` can be called from any component.

---

### Requirement 12: Protected API Endpoint Migration

**User Story:** As a backend operator, I want `/api/follows/*` and `/api/push-tokens/*` to derive user identity from the JWT `sub` claim so that clients cannot impersonate other users by supplying a different `userAddress`.

#### Acceptance Criteria

1. THE `FollowsController` and `PushTokensController` SHALL no longer accept a client-supplied `userAddress` to identify the caller; instead they SHALL read the identity from the `authenticatedAddress` request attribute set by `AuthFilter`.
2. WHEN the JWT `sub` is a UUID (email-auth user), THE controllers SHALL look up the user's `wallet_address` from the `users` table and use it as the `user_address` for follow and push-token operations.
3. WHEN the JWT `sub` is a wallet address (wallet-signature-auth user, matching `^0x[0-9a-fA-F]{40}$`), THE controllers SHALL use it directly as the `user_address`, preserving backward compatibility.
4. IF `AuthFilter` has not set `authenticatedAddress` (meaning the request reached the controller without a valid JWT, which should not happen given the filter), THE controllers SHALL return HTTP 401.
5. THE existing `AuthFilter` SHALL be extended to also protect `GET /api/auth/me` — adding it to the `PROTECTED` set so it requires a valid JWT.

---

### Requirement 13: Database Schema

**User Story:** As a backend operator, I want the new database tables created automatically on server startup.

#### Acceptance Criteria

1. THE `schema.sql` file SHALL include `CREATE TABLE IF NOT EXISTS users` with columns: `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`, `email TEXT NOT NULL UNIQUE`, `password_hash TEXT NOT NULL`, `wallet_address TEXT`, `email_verified BOOLEAN NOT NULL DEFAULT FALSE`, `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`.
2. THE `schema.sql` file SHALL include `CREATE TABLE IF NOT EXISTS email_verification_codes` with columns: `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`, `user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE`, `code VARCHAR(6) NOT NULL`, `expires_at TIMESTAMPTZ NOT NULL`, `consumed_at TIMESTAMPTZ`, `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`.
3. THE `schema.sql` file SHALL include an index `CREATE INDEX IF NOT EXISTS idx_evc_user_id ON email_verification_codes (user_id, consumed_at, expires_at)` for efficient code lookup.
4. THE `schema.sql` file SHALL include an index `CREATE INDEX IF NOT EXISTS idx_users_email ON users (email)` for efficient login lookups.

---

### Requirement 14: Security Constraints

**User Story:** As a security-conscious operator, I want the authentication system to follow standard security practices.

#### Acceptance Criteria

1. Passwords SHALL be hashed with bcrypt at a cost factor of at least 10 before storage. The plaintext password SHALL be cleared from server memory as soon as hashing is complete and SHALL NEVER appear in logs.
2. The same error response (`{ "error": "Invalid credentials" }` / HTTP 401) SHALL be returned whether the email does not exist or the password is wrong, preventing email enumeration on the login endpoint.
3. The same error response (`{ "error": "Invalid or expired code" }` / HTTP 400) SHALL be returned whether the code is wrong, consumed, expired, or the email does not exist, preventing email enumeration on the verify-email endpoint.
4. Verification codes SHALL be 6 digits generated by `SecureRandom` (server-side), single-use, and expire after 10 minutes.
5. THE Email_Auth_Service SHALL use the same `JWT_SECRET` environment variable and HS256 algorithm as the wallet-signature-auth service; no second secret is introduced.
6. THE resend-code endpoint SHALL enforce a rate limit of 3 requests per email per hour using an in-memory sliding-window counter (same pattern as `RateLimitService`).
7. SendGrid calls SHALL be made asynchronously on a separate thread (e.g., `CompletableFuture.runAsync`) so that email delivery latency does not block the HTTP response.
8. The `SENDGRID_API_KEY` environment variable SHALL be the only mechanism to configure the SendGrid API key; it SHALL NOT be hardcoded.
